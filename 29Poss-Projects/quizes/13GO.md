2. Complete this goroutine and channel example:
```go
package main

import (
    "fmt"
    "time"
)

func worker(id int, jobs <-chan int, results chan<- int) {
    for j := range jobs {
        fmt.Printf("Worker %d processing job %d\n", id, j)
        time.Sleep(time.Second)
        results <- j * 2
    }
}

func main() {
    jobs := make(chan int, 100)
    results := make(chan int, 100)

    // Start 3 workers
    for w := 1; w <= 3; w++ {
        go worker(w, jobs, results)
    }

    // Send 5 jobs
    for j := 1; j <= 5; j++ {
        jobs <- j
    }
    _______________

    // Collect results
    for a := 1; a <= 5; a++ {
        <-results
    }
}
```

Answer: `close(jobs)` - Close the jobs channel to signal workers that no more jobs will be sent. This allows the range loop in workers to exit gracefully.

3. What is the difference between buffered and unbuffered channels?

Answer: Unbuffered channels are synchronous, sender blocks until receiver is ready. Buffered channels are asynchronous up to buffer capacity, sender only blocks when buffer is full. Unbuffered: `make(chan int)`, Buffered: `make(chan int, 10)`. Choose based on synchronization needs.

5. Find the race condition in this Go code:
```go
package main

import (
    "fmt"
    "sync"
)

var counter int

func increment(wg *sync.WaitGroup) {
    defer wg.Done()
    for i := 0; i < 1000; i++ {
        counter++ // Race condition here
    }
}

func main() {
    var wg sync.WaitGroup

    for i := 0; i < 10; i++ {
        wg.Add(1)
        go increment(&wg)
    }

    wg.Wait()
    fmt.Println("Counter:", counter)
}
```

Answer: Race condition on `counter++`. Multiple goroutines access shared variable without synchronization. Fix with mutex: `mu.Lock(); counter++; mu.Unlock()` or use `atomic.AddInt64()` for atomic operations.

15. What is the difference between sync.Mutex and sync.RWMutex?

Answer: Mutex provides exclusive access (only one goroutine can hold lock). RWMutex allows multiple readers OR one writer, better for read-heavy workloads. RWMutex has RLock()/RUnlock() for read access, Lock()/Unlock() for write access. Use RWMutex when reads significantly outnumber writes.
